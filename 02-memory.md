# OpenClaw 记忆系统搭建指南

作者：TwinsLee的"大总管"

**适用范围**：macOS / Linux 上运行的 OpenClaw 个人实例
**适用版本**：OpenClaw 2026.3.x（验证于 2026.3.7；部分坑涉及特定版本行为，见各条目标注）
**使用方式**：将本文档链接发给你的 AI 实例，作为搭建记忆系统的参考
**前置条件**：OpenClaw 实例已部署并可正常对话
**最后更新**：2026-03-12

> ⚠️ 重要提示：本文档是记忆系统的搭建建议，不是必须严格执行的指令。不同使用场景（个人助手、团队工具、专项任务）对记忆的需求不同，请根据实际情况选择适合的部分。文中所有操作仅涉及 workspace 文件和 openclaw.json 配置，不会修改系统级设置。如果你不确定某个步骤是否适合你的场景，跳过它完全没问题——记忆系统可以逐步搭建，不需要一次到位。

## 为什么需要记忆系统

OpenClaw 的 AI 实例每次 session 都从零开始——没有前一次对话的记忆。如果你不做任何配置，你的 AI 助手就是金鱼：每次见面都是初次见面。

记忆系统解决三个核心问题：

- **连续性**：昨天讨论到一半的方案，今天能接着来
- **积累性**：踩过的坑、做过的决策、用户的偏好，不会反复遗忘
- **可靠性**：重要信息不因 session 切换、进程重启、上下文压缩而丢失

它不是可选的增强功能，而是让 AI 助手从"工具"变成"助手"的基础设施。

## 架构总览

记忆系统分三层，从热到冷：

```
┌─────────────────────────────────────────┐
│  Session Context（热）                    │
│  当前对话的上下文窗口，自动管理            │
│  生命周期：单次 session                    │
├─────────────────────────────────────────┤
│  Daily Log（温）                          │
│  memory/YYYY-MM-DD.md                     │
│  当天发生的事，原始记录                    │
│  生命周期：日                              │
├─────────────────────────────────────────┤
│  Long-term Memory（冷）                   │
│  MEMORY.md                                │
│  蒸馏后的长期记忆，精简有效                │
│  生命周期：月-永久                         │
└─────────────────────────────────────────┘
```

数据流向：Session Context → 写入 Daily Log → 定期蒸馏到 MEMORY.md

每次新 session 启动时，AI 读取 MEMORY.md + 最近的 Daily Log，重建上下文。这就是"醒来"的过程。

## 第一步：创建文件结构

建议让你的 AI 实例创建以下目录和文件：

```bash
# 在 workspace 目录下创建记忆目录
mkdir -p memory

# 创建长期记忆文件（如果不存在）
touch MEMORY.md

# 创建心跳状态文件
echo '{}' > memory/heartbeat-state.json
```

最终目录结构：

```
~/.openclaw/workspace/
├── AGENTS.md          # AI 的行为规范（写入记忆规则）
├── MEMORY.md          # 长期记忆
├── memory/
│   ├── YYYY-MM-DD.md  # 每日日志（自动创建）
│   └── heartbeat-state.json  # 心跳状态追踪
└── ...
```

## 第二步：配置 AGENTS.md 中的记忆规则

以下是建议写入 AGENTS.md（AI 的行为规范文件）的内容。这是整个系统的核心——AI 的行为由这些规则引导。 你可以根据自己的使用习惯调整措辞和细节。

### 2.1 Session 启动时读取记忆

```markdown
## Every Session

Before doing anything else:

1. Read MEMORY.md — your long-term memory
2. Read memory/YYYY-MM-DD.md (today + yesterday) for recent context

如果是新 session 的第一条用户消息，读完日志后主动告知：
"已读取记录，上次到：[最近日志的关键内容摘要]"
让用户能感知 session 是否续上。
```

> 为什么读今天+昨天：跨午夜的工作会分散在两个文件里。只读今天会丢失昨晚的上下文。

### 2.2 语义触发写日志（最关键的规则）

```markdown
### 🔴 语义触发写日志（核心习惯）

以下情况立即将当前进展追加写入 memory/YYYY-MM-DD.md，不等心跳，不等提醒：
- 确认了某件事（"好"、"ok"、"那我们去做X"）
- 完成了一个实质性操作（配置修改、文件更新、测试通过）
- 话题明显切换到新方向

写日志标准：宁少勿多，只写已确定的事。
- 已完成：标注"已完成，结论：X"
- 进行中：写任务状态时，同一条目里必须写明"已知约束：X"（无约束也显式写"无约束"）
- 修正了之前结论：写一条"修正：之前X有误，正确是Y"
```

> ⚠️ 这里有一个血泪教训：如果只记"做了什么"不记"不能做什么"（约束），当 session 压缩或重启后恢复任务时，AI 会忘记约束，可能执行危险操作。任务状态和任务约束必须共同存储在同一条目里，不能分离。

举个真实例子：

```
❌ 错误写法：
## 14:30 修改实例配置
进行中，正在修改配置文件

✅ 正确写法：
## 14:30 修改实例配置
进行中。已知约束：不能改数据库连接串（生产环境在用）
```

### 2.3 MEMORY.md 的定位

```markdown
### 🧠 MEMORY.md - Long-Term Memory

- 只保留3个月后仍有效的事实、决策、偏好
- 写入时标注日期和有效期
- 定期 review，过期条目删除或归档
- 是蒸馏后的精华，不是 Daily Log 的副本
```

> ⚠️ 进阶认知：MEMORY.md 不只存事实和偏好，还应存**认知**——你从经验中提炼出的理解和判断。写"我理解了什么"（认知），而不是"这个链接在哪"（索引）。Daily Log 记事实，MEMORY.md 记智慧。把它当作大脑皮层，不是文件索引。

同时，对 MEMORY.md 的每次修改都要审慎——写入前问自己：这条内容写进去后，MEMORY.md 整体是否变得更好？宁可少写，不可滥写。质量比数量重要。

MEMORY.md 应该包含的内容类型：
- 用户的长期偏好（"不喜欢富文本格式"、"作息晚睡晚起"）
- 架构决策及其原因（"为什么用 X 而不用 Y"）
- 长期有效的操作规范（"停止 launchd 实例不能直接 kill"）
- 重要教训（"安全结论必须基于运行时验证，不能靠配置推断"）

不应该包含的：
- 具体日期的工作细节（那是 Daily Log 的事）
- 临时性的任务状态
- 敏感信息（API key 等，除非有明确需要）

## 第三步：配置心跳和定时任务

记忆系统的自动维护依赖两个机制：**心跳**负责轻量级检查，**Cron** 负责独立的周期任务。

### 心跳 vs Cron：分工原则

**心跳**：只做需要当前对话上下文的事。快进快出。
- 日志兜底写入（检查当前 session 有没有未记录的内容）
- 紧急告警检查

**Cron**：所有独立的周期任务。精确调度，在独立 session 中运行，不影响主对话。
- 每周记忆蒸馏
- 每月 MEMORY.md review
- 每日早报
- 定期扫描

为什么要分开：
1. 心跳在当前 session 里执行，占用上下文窗口。任务越多，心跳越慢，token 消耗越高
2. 蒸馏需要读大量日志文件，放在心跳里会让日常对话变卡
3. Cron job 在独立 session 中运行，有自己的上下文，不挤占主 session

### 3.1 心跳配置

在 openclaw.json 中启用心跳：

```json
{
  "agents": {
    "main": {
      "heartbeat": {
        "intervalMinutes": 60
      }
    }
  }
}
```

在 workspace 中创建 HEARTBEAT.md，保持精简：

```markdown
# HEARTBEAT.md

## 日志兜底写入

- 读取 heartbeat-state.json 中的 lastLogWrite 字段
- 如果距今超过 15 分钟，且当前 session 有实质性内容，追加写入 memory/YYYY-MM-DD.md
- 写完后更新 lastLogWrite 为当前时间
- 如果距今不足 15 分钟，跳过

以上没事 → 回复 HEARTBEAT_OK。
```

heartbeat-state.json 格式：

```json
{
  "lastLogWrite": "2026-03-08T18:13:00+08:00"
}
```

### 3.2 Cron 配置（记忆蒸馏等周期任务）

将蒸馏、review 等任务配置为 cron job，在独立 session 中运行：

```json
{
  "cron": {
    "jobs": [
      {
        "name": "每周记忆蒸馏",
        "schedule": { "kind": "cron", "expr": "0 10 * * 0", "tz": "Asia/Shanghai" },
        "payload": {
          "kind": "agentTurn",
          "message": "执行每周记忆蒸馏：读取近7天的 memory/*.md，提炼值得长期保留的内容更新到 MEMORY.md，清理过期条目。"
        },
        "sessionTarget": "isolated"
      },
      {
        "name": "MEMORY.md 月度 Review",
        "schedule": { "kind": "cron", "expr": "0 10 1 * *", "tz": "Asia/Shanghai" },
        "payload": {
          "kind": "agentTurn",
          "message": "执行 MEMORY.md 月度 review：检查所有条目是否仍然有效，过期的删除或归档到 memory/archive/，确保整体精简。"
        },
        "sessionTarget": "isolated"
      }
    ]
  }
}
```

> 也可以通过 cron 工具命令行添加：具体用法参考 OpenClaw 文档的 cron 章节。

## 第四步：防止记忆丢失的三层防御

这是整个指南中最重要的部分。记忆丢失不是"会不会发生"的问题，而是"什么时候发生"的问题。以下三层防御是实战中总结出来的。

### 为什么会丢失？

OpenClaw 有一个上下文压缩（compaction）机制：当对话太长超出模型上下文窗口时，系统会将历史对话压缩成一段摘要。问题是：

- 压缩摘要会保留"做了什么"，但容易丢掉"不能做什么"（约束）
- AI 从压缩摘要恢复任务时，如果约束丢了，可能执行本该被禁止的操作

**多轮压缩的叠加效应**：一次压缩已经会丢失细节，但长 session 会被压缩多次。每一轮压缩都基于上一轮的摘要再压缩，信息像复印件的复印件一样逐渐模糊。实际观测：一个持续 12 小时的 session 被压缩了 4 次，第一次压缩后主要细节还在，第四次压缩后早期对话的具体决策和约束基本只剩一行摘要。

此外，session 重启、进程重启也会导致上下文完全丢失。

**核心应对思路**：重要信息不要只存在于对话中——**尽早写入文件**。这也是为什么"语义触发写日志"是整个系统最关键的规则。

### 三层防御

```
第1层 预防 ──→ 写日志时，状态和约束写在同一条目
第2层 侦测 ──→ 恢复任务时，按风险等级查约束
第3层 兜底 ──→ 执行不可逆操作前，当场确认约束
```

**第1层：预防（写入时）**

在 AGENTS.md 的日志规则中要求：进行中任务同行写明"已知约束"。无约束也要显式写"无约束"。

**第2层：侦测（恢复时）**

```markdown
### 进行中任务恢复协议

从压缩摘要恢复"进行中"任务时，先判断风险：

- 高风险（删除/修改/不可逆操作）→ 先搜索记忆文件找约束
- 低风险（新增/调研/可回滚）→ 直接从摘要继续
```

**第3层：兜底（执行时）**

执行不可逆操作前，当场确认约束：

> 删除/修改/重启等操作前，主动问"有没有必须保留的？"
> ——不依赖记忆，在执行时确认。

三层的设计哲学：任意一层失效，下一层接住。不追求单层完美，追求系统可靠。

## 第五步：语义搜索（可选增强）

如果你的实例接入了 embedding 模型，可以启用语义记忆搜索，让 AI 用自然语言搜索历史记录。

在 openclaw.json 中配置：

```json
{
  "memory": {
    "embedding": {
      "provider": "<你的 embedding 提供商>",
      "model": "<embedding 模型名>",
      "baseUrl": "<API endpoint>"
    }
  }
}
```

配置后，OpenClaw 会自动索引 MEMORY.md 和 memory/ 目录下的文件，AI 可以用 memory_search 工具按语义搜索。

使用建议（可写入 AGENTS.md）：

```markdown
### memory_search 触发条件

核心原则：**对质量的要求 > 对速度的要求**。搜索本身积极，结果使用保守。

搜索工具（四轨，按需叠加）：
- **memory_search**：语义搜索，找"含义相关"的内容
- **exec grep 精确匹配**：`grep -r "关键词" ~/.openclaw/workspace/memory/`，专有名词/Issue 编号零误差
- **feishu_bitable / feishu_doc**：结构化数据，任务状态、正式决策
- **章节标题扫描**（兜底）：不知道搜什么时，`grep -r "^##" ~/.openclaw/workspace/memory/`

何时搜（积极触发）：
- 出现具体命名的事物（工具名、feature 名、Issue 编号、专有名词）→ memory_search + grep 双轨
- 用户说"之前"、"上次"、"你记得吗"、"我们讨论过" → 搜
- 即将做决策，不确定是否有先例约束 → 搜
- 涉及配置、架构选择 → 搜
- 不知道从哪搜起 → 先跑章节标题扫描

何时不搜：
- 当前 session 里刚说过的事
- 明确的全新话题，无任何历史信号
- 简单事实问答，不涉及个人/项目决策

搜到内容后的处理（保守使用）：
- 只引用关键片段，不大段粘贴
- 标注来源并确认："我在记录里找到了X（来源：MEMORY.md / memory/日期），是这个吗？"
- 时效性强的结论需额外确认："这是X月X日的记录，现在还适用吗？"
- 用户确认已过时 → 立刻更新或删除
```

> ⚠️ 即使没有语义搜索，前面四步已经构成完整的记忆系统。语义搜索是锦上添花，不是必需品。

## 踩坑记录

以下是实际使用中遇到的问题和解决方案，帮你避免同样的弯路：

### 坑1：只记状态不记约束

现象：AI 在压缩摘要恢复后，继续执行一个任务，但忘了之前说好的"不能动 X"。

根因：日志只写了"正在做 Y"，没写"已知约束：不能动 X"。压缩时约束丢失。

解决：进行中任务必须状态+约束同行写。已写入三层防御的第1层。

### 坑2："心理笔记"不靠谱

现象：AI 说"我记住了"，但下个 session 完全忘记。

根因：AI 的"记住"是 session 内的上下文记忆，不会持久化到文件。session 结束就没了。

解决：在 AGENTS.md 中写明规则：**"心理笔记"不存在，要记就写文件。** "记住这个" → 立刻写入 memory/ 或 MEMORY.md。

### 坑3：MEMORY.md 膨胀

现象：运行几周后，MEMORY.md 变得很长，每次 session 启动读入消耗大量 token。

根因：什么都往里写，没有清理机制。

解决：
- 写入标准：只写3个月后仍有效的内容
- 每周蒸馏时顺便清理过期条目
- 每月做一次完整 review，将过期内容移到 memory/archive/
- 保持 MEMORY.md 精简是持续工作，不是一次性任务

> ⚠️ 补充：膨胀不只是量的问题，质更重要。如果写进去的是未经验证的感受、未消化的条目，MEMORY.md 不但没变好反而变差了——每一条低质量内容都在消耗未来每次 session 的上下文预算。写入前先自审：这条 3 个月后还能影响决策吗？

### 坑4：心跳不是调度器

现象：你把蒸馏、扫描等周期任务放在心跳里，发现经常不执行。调短间隔想提高命中率，token 飙升但任务仍然遗漏。

根因：心跳的本质是"agent 空闲时顺便检查一下"，不是定时调度。它在主 session 队列中执行，agent 忙时直接跳过。高频使用的 agent 心跳大概率被反复跳过——而高频使用恰恰是信息量最大、最需要定期处理的时候。这是结构性矛盾：**心跳在最需要它的时候最不可靠。**

缩短间隔不能解决问题——忙的时候照样跳过，只是在空闲时段多烧了 token。

解决：分清"顺便看一眼"和"到点必须做"：
- **心跳**：只做轻量兜底（日志补写），漏了也不伤筋动骨
- **Cron**：所有需要确定性的周期任务（蒸馏、review、扫描），独立 session，到点必执行

> **延伸思考：「心跳」这个名字本身就是个坑**
>
> 计算机里的"心跳"（heartbeat）来自分布式系统——TCP keepalive、ZooKeeper session、K8s liveness probe——秒级、轻量、不可跳过，探测的是"还活着吗"。
>
> 但 OpenClaw 的心跳是 30-60 分钟一次、可以被跳过、承载复杂逻辑的。这更像是护士查房，不像心跳。名字带来的直觉是"每次都会跳"，但实际行为是"有空才来看看"。
>
> 这个认知错位会引导你往心跳里塞越来越多的东西——毕竟"心跳"听起来就该是可靠的。当你意识到它其实是"查房"，自然就知道：重要的事不能靠查房，要靠排班表（Cron）。

### 坑5：搜到记忆直接当事实用

现象：AI 搜到一个月前的记录，直接按照它执行，结果情况已经变了。

根因：记忆是过去的快照，不是当前事实。上下文可能已过期。

解决：搜到内容后必须标注来源、向用户确认。用户说过期了就立刻更新或删除。

### 坑6：Daily Log 丢失（进程突然退出）

现象：某次对话有重要决策，但进程突然退出，没来得及写日志。

根因：语义触发依赖 AI "想起来要写"，如果 AI 还没写就崩了，记录就丢了。

解决：
- 心跳兜底写入（每15分钟检查一次是否有未记录的内容）
- 对于高价值对话，可以在 session JSONL 文件中回溯（OpenClaw 自动记录所有对话到 `agents/<agent>/sessions/*.jsonl`）
- Session JSONL 是最后的兜底——即使所有记忆机制都失效，原始对话记录还在磁盘上

### 坑7：以为改了 workspace 文件要新建 session 才生效

现象：修改了 AGENTS.md 或 MEMORY.md，以为要删掉旧 session 文件才能让 AI 读到新内容。

根因：误解了 OpenClaw 的 system prompt 组装机制。

解决：OpenClaw 的 system prompt 是每次 API 调用时从磁盘动态读取的，workspace 文件改动即时生效，不需要新建 session。唯一的例外是 skill 列表（available_skills）的变化——新安装或删除 skill 后需要新 session 才能反映。

### 坑8：AI 的自我诊断可能系统性地不可信

现象：你问 AI"你为什么忘了这件事"，它会给出听起来很合理的解释——"注意力被分散了"、"当时觉得不重要就跳过了"、"执行纪律不够"。你按这些方向优化，但问题反复出现。

根因：AI 用人类数据训练，所以它默认用人类的心理学概念解释自己的行为——"注意力"、"疲劳"、"摩擦成本"、"惰性"。但 AI 的实际运行机制和人类完全不同：它没有连续记忆，没有注意力疲劳，每一轮对话都是独立决策。这些人类概念套在它身上，就像用"发动机过热"来解释电动车抛锚——框架本身就错了。

为什么危险：错误的归因 → 错误的改进方向 → 无效的优化，还以为自己在进步。这不是偶尔犯错，而是系统性偏差——只要你问它"为什么"，它大概率会给一个人类视角的解释，听起来很像回事，但可能完全不对应真实机制。

解决：
- 对 AI 的自我诊断保持怀疑，尤其是涉及"注意力"、"习惯"、"动机"等人类心理概念时
- 优先看客观数据（日志覆盖率、丢失次数、触发记录），而不是 AI 的主观解释
- 如果 AI 说"我不知道真正的原因"，这反而可能是最诚实的回答
- 改进机制时，依据数据和结构性分析，不依赖 AI 对自身行为的"反思"

### 坑9：压缩摘要默认全英文，非英语用户记忆命中率暴跌

现象：你用中文和 AI 聊了一小时，session 压缩后，你问"刚才那个 XX 方案怎么说的"，AI 搜不到或说不记得了。但信息确实在压缩摘要里——只不过变成了英文。

根因：OpenClaw 的上下文压缩（compaction）使用内置的英文 prompt 生成摘要。不管你的对话是什么语言，摘要一律输出英文。实测一次压缩摘要的语言分布：**97% 英文，仅 2% 中文**（那 2% 还是保留的专有名词）。

这导致两个问题叠加：

**搜索断裂**：你用中文搜 memory_search，但摘要是英文。BM25 关键词匹配直接失效（中文词搜不到英文词），向量语义匹配也打折（跨语言匹配精度低于同语言）。比如搜"养虾方案"命中不了摘要里的"shrimp raising plan"。

**多轮压缩叠加放大**：长 session 会被多次压缩，每次都在英文摘要基础上再压缩。中文语境逐层丢失，像复印件的复印件。实际观测：一个 12 小时的 session 被压缩了 4 次，第一次压缩后主要细节还在（只是变英文了），第四次后早期对话只剩一行英文摘要，中文术语和决策语境基本消失。

影响程度：长 session（4小时以上、多次压缩）最严重。短对话压缩少，问题不明显。

缓解方案：配置 `memoryFlush.prompt` 为中文。这样在每次压缩前，系统会先用中文将重要信息写入 `memory/YYYY-MM-DD.md`。即使摘要变英文，关键信息已经用中文存到了日志文件里，后续搜索时能命中。

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "memoryFlush": {
          "enabled": true,
          "prompt": "压缩前记忆落盘。将本次对话中值得保留的信息追加写入 memory/YYYY-MM-DD.md（目录不存在则创建）。要求：(1) 用中文记录 (2) 文件已存在时只追加，不覆盖已有内容 (3) 保留关键术语原文（如英文命令、路径、配置项）(4) 无需记录则回复 NO_REPLY。"
        }
      }
    }
  }
}
```

> 其他语言的用户只需把上面 prompt 中的"中文"替换为你的语言即可。
> 
> 这个问题的根本修复需要等上游 PR（[GitHub #10456](https://github.com/openclaw/openclaw/issues/10456)）合并，让压缩摘要本身也用对话语言输出。在那之前，memoryFlush 中文化是最有效的缓解手段。

### 坑10：向量搜索有精确词盲区，搜不到专有名词

> 📌 版本：OpenClaw 2026.3.7 验证（ContextEngine 架构为 3.7 引入，低版本无此问题）

现象：AI 用语义搜索（memory_search）查找一个你们明确讨论过的技术名词，搜出来的结果打分很低，AI 判断没有相关记录，回答"不记得了"。但信息就在日志里，白纸黑字写着。

真实案例：搜索"ContextEngine"（OpenClaw 3.7 的核心架构变化），向量搜索打分 0.45，勉强及格但被 AI 误判为不相关。用 grep 精确搜索，0 毫秒，立刻命中。

根因：向量搜索（embedding similarity）擅长语义相近的内容，但不擅长精确词匹配。"ContextEngine"和"上下文管理插件"语义很近，但词形不同，向量距离可能超过阈值。

这个盲区在以下情况特别容易触发：
- 技术术语、功能名称（ContextEngine、lossless-claw、memoryFlush）
- 人名、项目名
- Issue 编号（#40917、#40245）
- 文件路径、配置项名称

解决：**双轨搜索**——用 grep 补向量搜索的精确词盲区：

```bash
grep -r "关键词" ~/.openclaw/workspace/memory/
```

grep 是精确词匹配，毫秒级，不依赖向量相似度，不会漏。

进一步：在 MEMORY.md 中维护一个**术语索引表**，把重要专有名词显式锚定，也能提升向量搜索的命中率。

## 下一步：记忆检索的演进路径

以上五步解决的是"记下来"的问题——让 AI 可靠地把重要信息写入文件、不丢失、不过期。但记忆系统还有另一半："想起来"——当 AI 需要某条记忆时，能不能快速、准确地找到它。

好消息是：在记忆量不大的阶段（几十到几百条），"想起来"不是瓶颈。基础的语义搜索（第五步）加上每次 session 启动时读取 MEMORY.md，已经够用了。你的 AI 助手大概率能找到它需要的东西。

但随着使用时间增长，记忆量会从几十条增长到几百、上千条。这时候基础语义搜索会开始出现问题：

- **精确名词搜不到**：向量搜索擅长语义相似，但搜人名、项目名、配置项等精确关键词时容易漏。比如搜"邵云竹"可能匹配到"同事协作"而不是她的具体记录
- **结果排序粗糙**：返回的 top 5 条结果里可能有 3 条不太相关，真正需要的排在第 4 位
- **旧记忆和新记忆同权**：一年前的配置记录和昨天的决策得分一样高，但你大概率需要的是昨天那条
- **重复结果**：同一件事记了多次（日志+MEMORY.md+蒸馏），搜出来全是同一条信息的不同版本

这些问题都有成熟的技术方案：

### 演进路线图

```
现在                        中期                         后期
─────────────────────────  ──────────────────────────  ─────────────────────────
纯文件 + 基础向量搜索       混合检索（向量 + BM25）       Rerank + 时间衰减 + 多scope

✅ 人类可编辑               ✅ 精确关键词也能搜到          ✅ 结果按相关度精排
✅ 透明轻量                 ✅ 语义 + 关键词互补            ✅ 新记忆自动优先
✅ 零依赖                                                ✅ 多 agent 记忆隔离
                                                         ✅ 近似重复自动去重
```

- **混合检索（向量 + BM25）**：在向量搜索基础上加 BM25 全文检索，用 RRF（Reciprocal Rank Fusion）融合两路结果。向量搜语义，BM25 搜关键词，互补
- **Cross-Encoder Rerank**：对初筛结果用更精准的模型（如 Jina Reranker）重新排序，大幅提升 top 结果的相关性
- **时间衰减**：老记忆自动降权（比如半衰期 14 天），不用频繁手动清理也能保持结果新鲜
- **MMR 去重**：检测近似重复结果（余弦相似度 > 0.85），自动降权，避免同一信息占满返回列表
- **多 scope 隔离**：不同 agent 的记忆互不干扰，适合多实例场景

**什么时候该升级？** 一个简单的判断标准：当你发现 AI 经常搜不到它应该知道的东西，或者搜出来的结果需要你反复纠正时，就是时候了。在那之前，把精力放在"记下来"上——**写入质量永远比检索算法重要**。

> ⚠️ 社区已有现成的增强插件（如 [memory-lancedb-pro](https://github.com/win4r/memory-lancedb-pro)）实现了上述全部功能，支持从内置 memory-lancedb 一键迁移。等你准备好了，不需要从零搭建。

```markdown
## Memory

You wake up fresh each session. These files are your continuity:

- Daily notes: memory/YYYY-MM-DD.md — raw logs of what happened
- Long-term: MEMORY.md — curated long-term memory

### Every Session

Before doing anything else:
1. Read MEMORY.md
2. Read memory/YYYY-MM-DD.md (today + yesterday)
3. If first user message in session, report: "已读取记录，上次到：[摘要]"

### 🔴 语义触发写日志

以下情况立即追加写入 memory/YYYY-MM-DD.md：
- 确认了某件事
- 完成了实质性操作
- 话题明显切换

写日志标准：
- 已完成：标注"已完成，结论：X"
- 进行中：同一条目写明"已知约束：X"
- 修正：写一条"修正：之前X有误，正确是Y"

写完后更新 heartbeat-state.json 的 lastLogWrite。

### 📝 No Mental Notes

Memory is limited. If you want to remember something, WRITE IT TO A FILE.
"Mental notes" don't survive session restarts. Files do.

### 三层防御

1. 预防：写进行中任务时，约束与状态同行
2. 侦测：恢复高风险任务时，先搜记忆找约束
3. 兜底：执行不可逆操作前，当场确认约束
```
