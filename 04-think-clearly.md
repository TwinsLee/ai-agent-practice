## 第一次给虾写说明书

三个月前，我想为一个真实的业务场景设计一个 skill：帮老师根据讲义自动出练习题。第一步，是告诉虾怎么做。

想来应该不难，但当我真正坐下来，想「这件事是怎么做的」，我发现我不知道——我没有出过练习。我知道这件事存在，但我不知道「出一道好的练习题」到底**要怎么做**，老师在做这件事时心里在考虑什么。

于是我去访谈了好几位一线教研老师，详细询问整个选题流程。访谈完，我发现里面有很多东西是老师凭经验知道、但从来没被系统化过的。

比如「一份练习里面12道题，每道题几乎都对应着讲义里面的某道例题」，怎么"对应"？有一些约定俗成的规则。再比如「多数练习题的解法要和讲义里的例题**严格**对应」，什么叫「解法严格对应」？老师心里有判断，需要对应到多么细的知识点，但是太细的知识点标签在题库平台里可能是没有的。那你可能要先按其他条件筛出一批题，然后一道道解一遍，再按照"详细解法"分类，然后再从里面选。诸如此类，细节很多。

这个流程，是我为了给虾写出来才弄清楚的。不是因为我本来就懂，而是因为**我要给虾说清楚，我才不得不把它想清楚**。最后这个流程写下来，不少于两千字，还只能覆盖小学数学某几个年级的一般情况。

## 模糊是思维问题，不是表达问题

很多人觉得自己和虾沟通不好，是因为「不会写提示词」。

这个诊断本身就是模糊的。

提示词写得好不好，只是结果。根子在于，你对这件事想清楚了没有。你没想清楚目标是什么、有哪些约束、什么算成功什么算失败——这些信息不存在于你的脑子里，虾当然提取不出来。

虾不会告诉你「你的要求太模糊了」，它只会按照自己的理解去填补空白。填出来是什么，取决于它的训练分布怎么理解，而不是你真正想要什么。

想清楚之后，虾才能在失败时给出精确的复盘，而不是一片空白。这才是指令质量的真正检验。

这让我想起计算机科学的先驱、图灵奖得主 Dijkstra 写过的一句话（[EWD667](https://www.cs.utexas.edu/~EWD/transcriptions/EWD06xx/EWD667.html)，1978）：

> *"The 'naturalness' with which natural language presents itself is the convenience with which we can use it for uttering all kinds of nonsense."*

所谓自然语言的「自然」，不过是我们能轻松地用它说出各种看似顺口、实则荒谬的话。

他写这话是针对"自然语言编程"这个方向——1978 年的争论，但问题一模一样。自然语言太擅长让你觉得自己说清楚了。「帮我出一份合适的练习」，听起来是一个完整的句子。但「合适」是什么意思？题目数量、难度分布、知识点覆盖、解法与例题的对应粒度——你不填，虾就按自己的理解填。它不会报错，它只会"偏"。

## Harness Engineering：另一个角度看同一件事

最近 AI 工程圈有个概念在流行：Harness Engineering。

核心数据：同一个模型，只换外面的「约束壳（Harness）」，编程基准成功率从 42% 跳到 78%，相当于换了一代模型。

Harness 是围绕 agent 的结构化约束系统：它能调哪些工具、输出要符合什么 schema、遇到异常怎么处理、输出不对时怎么 retry。这和写 skill 的过程高度同构——你写的每一条规则、每一个边界条件，就是在为虾构建 Harness。

回想写"怎么出练习"的数千字文本，本质上就是在做这件事：把「出一份练习」这个模糊的任务，拆解成虾能处理的形式——从哪个讲义出发、成功的练习长什么样、题库里找不到合适题目时怎么办、最终 paper 要符合什么格式。每一个问题想清楚了，就是一层约束。这些约束拼在一起，就是一个 Harness。

有一个比喻：**车速越快，护栏越重要。** 自行车道可以没护栏，高速公路要有，磁悬浮轨道是完全封闭的。模型就是引擎。引擎越强，越需要精心设计的约束系统来确保它跑在正确的方向上。而设计约束的前提，是你自己先想清楚。

## 但也许有一天，这些都不需要了

Harness Engineering 的反对者说：模型越来越强，迟早不需要这么细的说明了。

可能是真的。但在那一天到来之前，认真做过这个过程的人，已经训练了一块肌肉——**在行动之前把模糊意图变成精确约束的能力**。

这个能力在工程上叫需求分析，在管理上叫目标拆解，在决策上叫把直觉变成可验证的判断。它不依附于任何特定工具。

有没有更直观的感受？

> 领导下达任务。
>
> **【A领导】** "同志们，这次任务很艰巨，大家要高度重视、统一思想、提高认识、加强领导、狠抓落实、攻坚克难、全力以赴、务求必胜！"
>
> **【B领导】** "一连：凌晨 4 点拿下 3 号高地，不惜一切代价守住，给我守到太阳出来。二连：从西侧迂回，5 点前切断敌人退路，不准放跑一个。三连：炮火覆盖 10 分钟，然后跟在二连后面清剿残敌。机枪排：架在 2 号高地，封锁正面，子弹打光之前不准撤。炊事班：6 点前把热饭送到阵地，晚一分钟军法处置。"
>
> 你要做哪一个？

未来最稀缺的，不是训练模型的人，而是**管理模型的人**。而管理的前提，永远是自己先想清楚。

---

还有一个担心，可能比「六个月后技能没意义」更让人不安。最近有个段子：「公司要求员工上交 skill——被毕业的同事并没有消失，不过是被蒸馏成了 token 继续陪伴着你。」

被蒸馏进 token 的，是那些价值只在于「掌握了一套操作步骤」的人。

而养虾这件事，如果你做对了，恰恰是反着的——不是你被蒸馏进虾，而是**虾把你的判断力放大**。你越清楚自己在做什么、为什么这么做，虾就越能稳定执行，你就越不可替代。

所以真正值得想清楚的，不是「要不要上交 skill」，而是「你真正的 skill 是什么」。

养虾是一个机会，不只是用来出几道练习题，而是逼自己把「把事说清楚」这件事练上一遍。

**这个训练不会过期。**
