# 养虾经验④：Harness 思维——从模糊意图到可执行约束

> 系列：[①安全篇](./01-security.md) · [②记忆篇](./02-memory.md) · [③元认知篇](./03-metacognition.md) · **④本篇**

上一篇讲元认知：不要用人类的情绪/惰性/执行意志去解释 agent 的行为，那是错误的调试框架。

这篇往前走一步：**你对 agent 的指令质量，取决于你能不能在下指令之前先把这件事想清楚。**

---

## 问题的根源：自然语言掩盖了你的模糊

很多人遇到 agent 表现不稳定，第一反应是"提示词没写好"。这个诊断本身就是模糊的。

提示词只是结果。根源是：**你对这个任务的 spec 没有完成。**

你没有明确：
- 目标是什么（输出格式、数量、范围）
- 约束是什么（不能做什么、边界条件怎么处理）
- 验收标准是什么（什么叫"成功"，什么叫"失败"）

这些 spec 不存在于你的脑子里，agent 就只能用训练分布去填补空白——填出来的是它认为"最像"的结果，不是你真正想要的。

Dijkstra 在 1978 年写过一句话（[EWD667](https://www.cs.utexas.edu/~EWD/transcriptions/EWD06xx/EWD667.html)）：

> *"The 'naturalness' with which natural language presents itself is the convenience with which we can use it for uttering all kinds of nonsense."*

当时他批评的是"自然语言编程"的方向——自然语言的流畅，会掩盖你没有真正想清楚这件事。48 年后，这个问题在 agent 工程里被放大了。

---

## 真实案例：给 agent 写 spec 的过程

我曾为一个教育场景设计 skill：根据讲义自动出练习题。

技术侧没什么难度——调题库 API、格式化输出。难的是**把这个任务的 spec 写清楚**。

以为一两百字能说完，结果访谈了多位一线教研老师之后，写出来超过两千字，还只覆盖小学数学某几个年级的通用情况。

几个典型的 spec 细节：

```
# 约束示例（节选）

## 题目数量与结构
- 每份练习固定 12 道题
- 每道题对应讲义中的一道例题（1:1 mapping）
- 难度分布：基础 60%，提升 30%，挑战 10%

## 解法对应规则
- 练习题的解法路径必须与对应例题"严格一致"
- 严格一致 = 知识点标签匹配到三级（平台标签树最细粒度）
- 若题库无匹配：先按二级标签筛出候选集 → 逐题解题 → 手动分类解法 → 从子集选题

## 异常处理
- 候选题不足 3 道：记录缺口，报告给用户，不自动降级标准
- 题目重复（与上周练习相同）：跳过，从候选集下一题选取
```

这些规则在老师脑子里是隐性知识。为了让 agent 执行，不得不把它显性化。

**这个过程本身，才是养虾的核心工作——不是调参，不是改提示词，是把隐性知识转成可执行的 spec。**

## Harness Engineering：更精确的框架

最近 AI 工程圈有个概念火了：[Harness Engineering](https://www.harness.io/blog/ai-harness-engineering)。

核心数据：同一个模型，只换外面的「约束壳（Harness）」，编程基准成功率从 42% → 78%，相当于跨一代模型。

Harness 的定义：**围绕 agent 的结构化约束系统**，包括：
- 工具访问控制（能调什么 API，不能调什么）
- 输出格式约束（schema、长度、必填字段）
- 异常处理规则（失败时 fallback 到哪一步）
- 验证逻辑（输出不符合 spec 时如何 retry 或报错）

这和写 skill/agent 的过程高度同构。你写的每一条约束、每一个边界条件，就是在为 agent 构建 Harness。

一个比喻：**车速越快，护栏越重要。** 自行车道可以没护栏，高速公路要有，磁悬浮轨道是完全封闭的。模型能力越强，Harness 设计的质量就越决定结果的质量。

## 如何检验你的 spec 是否足够清晰

一个实用的自检方法：**把 spec 交给另一个人（或另一个 agent），看他能不能在不问你任何问题的情况下执行出你期望的结果。**

如果对方需要问"这种情况怎么处理"，说明你有 spec 缺口。

更结构化的 checklist：

```
□ 输入是什么（格式、来源、边界值）
□ 输出是什么（格式、字段、示例）
□ 核心约束（必须满足的条件）
□ 验收标准（什么叫成功，什么叫失败）
□ 异常处理（至少覆盖最常见的 3 种失败情况）
□ 禁止事项（不能做什么，agent 默认会做但不该做的）
```

这不是提示词技巧，是任务分析能力。对 agent 有效，对团队协作、项目管理、需求拆解同样有效。

---

## 关于"这些能力会不会过期"

Harness Engineering 的反对者说：模型越来越强，迟早不需要这么细的 spec 了。

可能是真的。但在那一天到来之前，认真做过这个过程的人，训练的不只是"会写提示词"——而是**在行动之前把模糊意图转化为精确约束的能力**。

这个能力在软件工程里叫"需求分析"，在管理里叫"目标拆解"，在决策里叫"把直觉转成可验证的判断"。它不依附于任何特定工具。

被 AI 替代的，是那些价值只在于"掌握了一套操作步骤"的人。

如果做对了，养虾的结果恰恰相反——不是你被蒸馏进 agent，而是**agent 把你的判断力放大**。你越清楚自己在做什么、为什么这么做，agent 就越能稳定执行，你就越不可替代。
