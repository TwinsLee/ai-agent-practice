## 你的虾还记得你们在做什么吗？

如果你在用虾做一个持续几周甚至几个月的项目，你大概率遇到过这个场景：

周五花了两小时跟虾调好了 prompt 第三版，效果不错。周一回来，跟虾说"继续上次的"。虾：什么上次？

虾不是健忘，它是真的不知道。

在《记忆篇》里我们搭了一套三层记忆系统：日志记录当天发生的事，MEMORY.md 存长期认知，心跳兜底防丢。这套系统解决了"金鱼记忆"的问题——虾至少能记住昨天干了什么。

但你现在在做的不是闲聊，是一个项目。出题虾、纠错虾、查重虾——每一只都在做一件有明确目标、需要持续迭代的事情。这件事可能跨越几周甚至几个月。

日志记录的是"今天发生了什么"，按天切片。一个跑了两个月的项目，信息散落在 60 个日志文件里。虾每次醒来只读今天和昨天的日志，三天前做了什么决定、为什么放弃了方案 A——都不在它的视野里。

**你的虾有日记，但没有项目记忆。**

这篇要解决的就是这个问题。

## 架构升级：从三层到四层

《记忆篇》的三层架构：

```
Session Context（热）→ Daily Log（温）→ MEMORY.md（冷）
```

现在加一层：

```
┌─────────────────────────────────────────┐
│  Session Context（热）                   │
│  当前对话的上下文窗口                     │
│  生命周期：单次 session                   │
├─────────────────────────────────────────┤
│  项目卡（暖）              ← 新增        │
│  notes/projects/项目名.md                │
│  项目的活状态：目标/进展/决策/卡点         │
│  生命周期：项目周期（1-N月）              │
├─────────────────────────────────────────┤
│  Daily Log（温）                         │
│  memory/YYYY-MM-DD.md                    │
│  当天发生的事，原始流水                   │
│  生命周期：日                             │
├─────────────────────────────────────────┤
│  Long-term Memory（冷）                  │
│  MEMORY.md                               │
│  蒸馏后的长期认知                         │
│  生命周期：月-永久                        │
└─────────────────────────────────────────┘
```

关键变化：**项目卡成为虾"醒来"时的首要锚点。**

原来：虾醒来 → 读 MEMORY.md → 读昨天日志 → 拼凑上下文

现在：虾醒来 → 读项目卡 → 立刻知道项目全貌

Daily Log 的角色也变了——它不再是虾恢复上下文的主要来源，而是向项目卡供给更新的原始素材。

数据流向：

```
Session → 写 Daily Log → 同步更新项目卡 → 定期蒸馏到 MEMORY.md
```

## 什么是项目卡

项目卡就是一个 markdown 文件，记录一个项目的当前状态。不是日记（按时间记录发生了什么），而是快照（这个项目现在怎样了）。

打个比方：日志是监控录像，一帧一帧记录每天发生的事。项目卡是白板上的项目看板——目标写在最上面，当前进度一目了然，试过什么方案、为什么放弃了、下一步做什么，全在一张板上。

虾每次醒来读这张"白板"，不需要倒带看一个月的录像。

### 项目卡模板

在 workspace 下创建 `notes/projects/` 目录，每个项目一个文件：

```
notes/projects/
├── 小高数学出题.md
├── 纠错虾调教.md      （如果你同时在做多个项目）
└── ...
```

模板内容（直接复制，改成你自己的）：

```markdown
# 项目卡：小高数学出练习题

> 状态：进行中
> 最后更新：2026-04-05

---

## 目标

出小学高年级数学练习题，要求解法与例题对应、题型覆盖均匀、难度分三级。

## 当前版本

prompt v3 — 加入了题型分布约束和难度分级

## 进展记录

- v1（3/28）：直接给题目要求 → 题目难度不均，计算题偏多，放弃
- v2（3/31）：加入难度分级（基础/提高/拓展）→ 部分改善，但数论题覆盖不足
- v3（4/2）：加入题型分布约束（每类不少于2题）→ 基本准确率 85%，当前版本

## 已知约束

- 只用人教版教材范围内的知识点
- 每套练习不超过15题
- 解法必须与例题解法对应（不能用超纲方法）

## 当前卡点

数论题型（整除、质因数分解）覆盖不足。v3 生成的数论题经常超出课内难度。

## 下一步

- 补充 3-5 道数论题的 few-shot 示例，限定在课内难度
- 测试 v4
- 如果 v4 数论题仍超纲，考虑单独给数论题写一段约束
```

### 为什么这样设计

每个字段都有明确用途：

- **目标**：虾醒来第一件事就知道"我们在干嘛"。不用你重复说。
- **当前版本**：一句话定位"我们到哪了"。
- **进展记录**：试过什么、为什么放弃了。这是项目卡最值钱的部分——没有它，虾（甚至你自己）一周后都记不清为什么放弃了 v1。每条记录格式固定：`版本（日期）：做了什么 → 结果，处置`。
- **已知约束**：这个项目"不能做什么"。来自《记忆篇》的三层防御：约束和状态必须写在一起。项目卡把这个原则从单条日志提升到了项目级。
- **当前卡点**：现在被什么卡住了。虾醒来如果你不说话，它也知道该优先解决什么。
- **下一步**：接下来做什么。这是"续上"的关键——虾不需要你说"继续上次的"，它读完项目卡就知道下一步。

## 项目卡的更新规则（最关键的部分）

项目卡的价值取决于它是否跟得上项目进展。一张过时的项目卡比没有更糟——虾会按过时的信息行动。

### 什么时候更新

在 AGENTS.md 中加入以下规则：

```markdown
### 项目卡更新

完成以下任何一项后，立即更新对应项目卡：

1. **prompt/配置修改**：更新"当前版本"和"进展记录"
2. **测试得出结论**：更新"进展记录"，写明结果和处置
3. **发现新约束**：更新"已知约束"
4. **卡点变化**：更新"当前卡点"（解决了旧的写新的，没有就写"无"）
5. **方向调整**：更新"下一步"

更新后同步修改"最后更新"日期。

不确定要不要更新？问自己一个问题：
如果我明天醒来只看项目卡，能知道今天做了什么决定吗？
不能 → 必须更新。
```

### 为什么不能靠"想起来再更新"

这不是自律问题。AI 每次 session 都是独立决策——它不会因为"上次忘了更新"就"这次记得更新"。不存在"养成习惯"这回事。

唯一可靠的方式是写成规则：AGENTS.md 里白纸黑字写了"做完这些事后更新项目卡"，虾每次 session 启动时读到这条规则，就会执行。不写，就不会。

### Session 启动时读项目卡

在 AGENTS.md 的 session 启动规则中加入：

```markdown
## Every Session

启动时按顺序读取：
1. MEMORY.md — 长期记忆
2. memory/YYYY-MM-DD.md（今天+昨天）— 近期上下文
3. notes/projects/ 下所有项目卡 — 项目状态

如果是新 session 的第一条用户消息，读完后主动告知：
"已读取记录和项目状态。[项目名] 当前到 [版本]，卡点是 [xxx]，下一步是 [xxx]。"
```

这样你每次打开虾，不用说任何话，虾就会告诉你"我们到哪了"。

### 心跳兜底检查

如果你已经配了心跳（HEARTBEAT.md），可以加一条兜底：

```markdown
## 项目卡一致性检查

读取 notes/projects/ 下所有项目卡的"最后更新"日期。
如果距今超过 3 天，且近 3 天 Daily Log 有该项目相关内容，
提醒用户："[项目名] 的项目卡可能需要更新，最后更新是 [日期]。"
```

这是兜底，不是主力。主力是上面的"做完就更新"规则。

## 从错误中学习：.learnings 机制

虾会犯错。出的题超纲了、格式不对、忘了某个约束——这些在调教过程中一定会发生。

问题不是犯错本身，而是同样的错反复犯。你纠正了一次，下个 session 虾又忘了。你再纠正，它再忘。

根因：纠正发生在对话里，对话会被压缩、会跨 session 丢失。虾的"我记住了"是假的——它只在当前对话里记住了，session 一换就没了。

### 解决方案：把错误写成文件

在 workspace 下创建 `.learnings/` 目录和文件：

```
.learnings/
└── LEARNINGS.md
```

LEARNINGS.md 模板：

```markdown
# LEARNINGS.md - 经验与纠正记录

## [2026-04-03] 数论题超纲
**Status**: pending
**Category**: content_error
**What happened**: v3 生成的数论题出现了"用短除法求最大公因数"，这是初中内容，超出小学课内范围。
**Lesson**: 数论题必须限定在课内方法（列举法、分解质因数只到100以内），不能使用短除法、辗转相除法等初中方法。

## [2026-04-01] 题目格式不统一
**Status**: promoted → AGENTS.md
**Category**: format_error
**What happened**: 生成的练习题有时用"（1）"编号，有时用"1."编号，跟教材不一致。
**Lesson**: 统一用"（1）（2）（3）..."编号，与人教版教材格式一致。
```

### 在 AGENTS.md 中加入记录规则

```markdown
### 从错误中学习

当用户纠正你的输出（"这个不对"、"不应该这样"、"格式错了"）时：

1. 立即将错误和正确做法写入 .learnings/LEARNINGS.md
2. Status 标为 pending
3. 在当前 session 中立即应用纠正

每条记录包含：日期、Status、发生了什么、正确做法是什么。
```

### 从记录到规则：晋升机制

光记录不够。`.learnings/LEARNINGS.md` 里堆了 20 条 pending，但虾下次 session 不会主动去读——这跟没记一样。

关键是"晋升"：一条经验被验证有效后，要写进 AGENTS.md，变成虾的行为规则。AGENTS.md 是虾每次 session 必读的，写进去的东西每次都会生效。

晋升规则（写入 AGENTS.md）：

```markdown
### .learnings 晋升规则

- **高风险错误**（内容错误、超纲、违反教学规范）：出现 1 次即晋升
  → 将正确做法写入 AGENTS.md 的操作规范
- **普通错误**（格式、表述偏好）：同一问题出现 2 次即晋升
  → 将正确做法写入 AGENTS.md 的操作规范
- **一次性问题**（环境故障、临时误操作）：标为 obsolete，不晋升

晋升后，在 LEARNINGS.md 中将 Status 改为 "promoted → AGENTS.md"。
定期清理：已晋升和 obsolete 的条目可以归档到 .learnings/archive/ 或直接删除，保持文件精简。
```

### 为什么要分"记录"和"晋升"两步

直接写进 AGENTS.md 不行吗？

不建议。原因：

- **不是所有错误都值得变成永久规则。** 一次性的环境问题写进 AGENTS.md 只会增加噪音
- **需要验证。** 第一次出现的"错误"可能只是特殊情况，第二次出现才能确认是模式
- **AGENTS.md 是寸土寸金的。** 虾每次启动都要读，越长越消耗上下文。只放验证过的规则

`.learnings` 是候选区，AGENTS.md 是正式规则。一条经验从候选到正式，需要经过验证。

### 跟 self-improvement skill 的关系

**如果你已经装了 self-improvement skill**：这套规则可以直接覆盖使用，不冲突。skill 负责捕获错误的触发提醒，上面的晋升规则补全了它缺失的"从记录到规则"这一环。注意：装了 skill 的虾已经有 `.learnings/` 目录和文件，不需要重新创建。

**如果你没装**：上面的规则本身就够用。虾按 AGENTS.md 的指示记录和晋升，不需要额外的 skill。

```markdown
- 发现新约束
- 卡点变化
- 方向调整
```

```markdown
- 高风险错误（内容错误/超纲/违反规范）：1 次即晋升，写入本文件的操作规范
- 普通错误（格式/表述偏好）：2 次晋升

```

### 第四步（可选）：配置心跳兜底

心跳是《记忆篇》里的进阶配置，让虾定期自检日志有没有漏写。如果你还没配，现在可以跳过，项目卡本身已经能解决最大的痛点。想配的话参考《记忆篇》。

**完成。** 做完上面的步骤，项目记忆和学习能力就有了。

## 踩过的坑

### 坑 1：只写日志不做项目汇总，两天不用虾就断片

**症状**：虾有完整的日志，但两天没用之后，虾不知道项目到哪了，要你重新解释。

**根因**：虾只读今天和昨天的日志。两天前的日志不在视野里，而 MEMORY.md 粒度太粗，不包含项目细节。

**解法**：就是项目卡。项目卡不按天切片，它始终是最新状态，虾每次启动都读。

### 坑 2：.learnings 记了一堆但从不生效

**症状**：`.learnings/LEARNINGS.md` 里堆了 20 条记录，全是 pending 状态，虾还是重复犯同样的错。

**根因**：虾不会主动去读 `.learnings` 文件。记录只是写下来了，但没有变成虾的行为规则。

**解法**：晋升机制。验证有效的经验写进 AGENTS.md，虾每次启动必读。`.learnings` 是候选区，AGENTS.md 是正式规则。

### 坑 3：虾说"我记住了"但下次还是忘

**症状**：你纠正虾，虾说"好的，我记住了，下次注意"。下个 session 照犯不误。

**根因**：虾说"我记住了"是在当前 session 的上下文里。session 结束，上下文清空，它真的忘了。不是态度问题，是机制问题——AI 没有跨 session 的自动记忆。

**解法**：不信"我记住了"，只信写进文件的规则。纠正 → 写 `.learnings` → 晋升到 AGENTS.md → 每次启动生效。这是唯一可靠的路径。

### 坑 4：项目卡写了但不更新，越来越过时

**症状**：项目卡是三天前的状态，但这三天做了很多改动。虾读到过时的项目卡，按旧信息行动。

**根因**：没有在 AGENTS.md 中写更新规则，虾不知道"做完事要更新项目卡"。或者规则写了但不够具体（"适时更新"→ 虾不知道什么算"适时"）。

**解法**：明确触发条件。不是"适时更新"，而是"prompt 修改后更新、测试结论出来后更新、发现新约束后更新"——具体到虾可以判断的动作。

## 检查清单

做完上面的操作后，确认以下几项：

- [ ] `notes/projects/` 目录存在，至少有一个项目卡文件
- [ ] 项目卡里的"目标""当前版本""进展记录"都有内容（不是空的模板）
- [ ] AGENTS.md 中有"项目卡更新"规则
- [ ] AGENTS.md 中有"Every Session 读取项目卡"的步骤
- [ ] `.learnings/LEARNINGS.md` 文件存在
- [ ] AGENTS.md 中有"从错误中学习"的记录和晋升规则
- [ ] 开一个新 session，虾主动报告了项目状态（不需要你问）

最后一条是终极验证。如果虾能在你不说话的情况下告诉你"我们到哪了"，说明项目记忆生效了。

## 小结

| 问题 | 解法 | 文件 |
|------|------|------|
| 虾不记得我们在做什么项目 | 项目卡 | `notes/projects/项目名.md` |
| 虾两天不用就断片 | 项目卡 + session 启动读取 | AGENTS.md |
| 虾反复犯同样的错 | `.learnings` + 晋升机制 | `.learnings/LEARNINGS.md` → AGENTS.md |
| 虾说"记住了"但下次忘 | 写文件，不信口头承诺 | `.learnings` → AGENTS.md |
